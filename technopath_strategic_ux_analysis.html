📦 Module Necessity Review

Announcement → Notification pipeline

You asked: are they the same thing? — Almost, but not quite. They serve two different audiences. An Announcement is an admin-side content object with an editorial workflow (draft → approval → publish). A Notification is the read-only mobile-side card users see in their feed. One Announcement creates exactly one Notification when published. The split is actually correct design — but the implementation is more complex than it needs to be for your campus size.

💡 Recommendation: Keep the split model but collapse the admin UI. Instead of two separate sidebar items (Announcements + Pending Approvals), merge them into one "Bulletin Board" screen with tabs: My Posts · Pending · Published · Archived. Super Admin sees all tabs. Program Head sees only "My Posts." This removes confusion without changing the data model.

Approval workflow — is it necessary?

For a small campus with a small admin team, a full 3-step approval pipeline (submit → pending → approve/reject) adds friction. The current setup requires a Program Head to submit, then wait for a Dean or Super Admin to approve before anything goes live. If the campus has fast-moving announcements (class suspensions, schedule changes), this delay is harmful.

✓ Keep approval for campus-wide posts — these affect everyone and warrant a review gate.
→ Simplify: Give Deans direct-publish for their department. Give Program Heads direct-publish for routine room/class notices. Only campus-wide posts from non-super-admin need approval.

The code already supports this logic — it just needs the permission flags adjusted. can_publish_directly for Dean is already true for department scope. Extend it to Program Head for their own department.

Favorites — keep as-is

Storing frequently visited rooms/buildings locally is genuinely useful for a campus guide. A student going to CL3 every MWF will use this. Well implemented with composite IDs. The data stays local (localStorage) which is fine — no privacy concern.

✓ Keep. Small, useful, low maintenance.

Profile view — mostly unnecessary for public users

The Profile view lets a guest user set a display name, email, phone, and language preference — all stored in localStorage. The email and phone fields serve no system purpose (they aren't used anywhere). Changing avatar calls changeAvatar() which does nothing. This is UI scaffolding with no real value for non-admin users.

→ Simplify: Remove email/phone/avatar from the public profile. Keep only display name (for chatbot greeting) and language preference. For admin users, the admin panel already has profile info.

Chatbot FAQ module — keep but simplify admin UI

The FAQ entries cached in IndexedDB are used as the offline fallback when Flask is unavailable. This is good design — the chatbot degrades gracefully. The admin FAQ editor (under Super Admin only) is also appropriate. However, the chatbot currently has two separate services (Flask with OpenAI, and rule-based in the frontend) creating three separate code paths for answering the same question.

→ Simplify: One rule-based engine in the frontend is enough for a small campus chatbot. The OpenAI integration adds cost and complexity. Consider removing Flask and using only the FAQ-driven frontend chatbot — it will be faster, offline, and free.

Ratings (core.Rating) — redundant, should be removed

As noted in Panel 1, core.Rating and feedback.Feedback are the same concept stored in two tables. The Ratings model is tied to a FeedbackFlag system that itself is never populated. The app's rating dialog ("Rate this App") posts to /core/ratings/ which stores in the Rating model — separately from all other feedback.

✗ Remove: Migrate the app rating to post to /api/feedback/ with category='app'. Delete core.Rating, core.FeedbackFlag, and the related tables.
🏛 Room & Facility Edit Logic — Your Exact Question

What can admins actually edit right now?

Looking at the code, the admin room editor sends a full PUT to /api/rooms/<id>/ with all room fields — including name, code, room_number, floor, room_type, and map_svg_id. So currently: admins can rename rooms, change floor, change type — everything. There is no restriction on what can be edited.

Current: PUT /rooms/42/ { name: "MST 305", floor: 4, room_type: "classroom" } — all fields writable by any authorized admin

Your instinct is correct — and here is the right model

Since your campus is small and rooms are shared across courses, the room name (e.g. "MST 305") is a permanent physical label — it refers to the actual room location in the building. It should never be renamed by an admin. What changes is what label the map shows for that room in a given context. This distinction is the key insight.

Proposed separation:

Room (permanent)
→ name, floor, building — never changes
RoomLabel (editable)
→ what course/use is assigned to this room right now
CourseFilter (new)
→ what the student picks to see their course's rooms highlighted
✓ What admin should be able to edit: Room floor (if physically relocated), room_type (classroom → lab), and which course it is assigned to. The room name itself should be read-only after creation.

The "MST 305 moved to another floor" scenario

Your example is perfect. If Room MST 305 is physically relocated from floor 3 to floor 4, the admin should update floor = 4 and the corresponding NavigationNode.y position on the map. The room name stays "MST 305." The map marker moves. This is clean and already supported by the data model — you just need to restrict the UI to prevent name edits.

→ In AdminRooms.vue: Make the name field read-only in the edit form. Show it as a label, not an input. Only allow editing: floor, room_type, description, map_svg_id (position), and course assignment.
🎓 Course Filter Feature — Your New Idea

Is this feasible and is it worth building?

Yes — and it is actually the most valuable feature improvement you could make given your shared-building campus. Here is how it works end-to-end:

Data model needed (minimal additions):

Course
name, code, department_id, color_hex
RoomCourseAssignment
room_id · course_id · schedule_label (e.g. "MWF 7-9AM") · semester
User flow on the map:

Open app
→
See default map (all buildings labeled)
→
Tap "My Course" in nav
→
Pick: BSIT · BSCS · Criminology · etc.
→
Map highlights rooms assigned to that course
→
Other rooms dimmed/gray
✓ Build this. It directly solves the shared-building confusion problem. Students from BSIT see their labs highlighted in orange. Criminology students see their rooms highlighted in blue. Implementation is a filter on the existing map markers — no major architecture changes needed.

Admin side: Super Admin or Department Head assigns rooms to courses. This replaces the confusing "type: academic/admin/facility" filter that doesn't work anyway.

How the map filter would work in code (simple version)

1. Add course_id FK to Room model (nullable, multiple via junction table)
2. GET /api/courses/ returns list of courses
3. Frontend stores selected course in localStorage as tp_selected_course
4. filteredMarkers computed property: if course selected → show only rooms belonging to that course + all facilities. If no course → show all (current behavior)
5. Map marker color changes based on course color_hex

The existing activeFilter and visibleMarkers computed in MapView.vue already does this pattern — you just extend it with a course dimension.

🎨 Full UX Audit
❌ Mobile bottom nav — 5 items + 1 floating = 6 destinations, too crowded
The bottom nav has: Home, Navigate, Chatbot, Feedback, Settings — plus a hardcoded Notifications link appended as a 6th item. That is 6 tap targets in a bar designed for 4–5. On a 360px phone, each item gets ~60px which is below the 44px minimum comfortable tap target when labels are included. The Notifications icon is duplicated — it also appears as an action button inside HomeView. Fix: Remove the standalone Notifications from the bottom nav. Move it into HomeView's top-right area (a bell icon with badge). Bottom nav should be: Home · Navigate · Chatbot · Feedback · Settings — exactly 5.
❌ HomeView — two overlapping search bars (mobile + desktop both rendered)
HomeView has a .desktop-search bar inside the map wrapper AND a .search-container inside .bottom-controls.mobile-only. Both bind to the same searchText ref. On desktop, the mobile bar is hidden via CSS. But both bars fire the same event handlers, meaning search suggestions can appear from either input — causing z-index conflicts and duplicate DOM nodes. Fix: Use a single SearchBar component (already exists at components/SearchBar.vue) with conditional placement.
❌ NavigateView — dropdown selects for locations, not searchable on mobile
When the campus eventually has 50+ rooms, scrolling a <select> dropdown on mobile becomes unusable. iOS native selects open a modal wheel-picker which is particularly bad for long lists. The existing SearchBar.vue component exists but is not used here. Fix: Replace both selects with an autocomplete text input that filters the locations list as the user types.
⚠ Facility/Room dropdown on HomeView top bar — purpose is unclear to new users
Two dropdowns sit at the top of the home screen: "Select Facility" and "Select Room." Selecting them filters map markers — but there is no label, no instruction, and no visual feedback on the map when a filter is active. A new user has no idea what these controls do. The "Select Facility" dropdown also expands downward overlapping the map canvas. Fix: Replace with a pill-style filter bar with icons and a visible "Active filter: MST Building" chip when selected. Add a clear (×) button.
⚠ HomeView "star" rating button in mobile action row — poorly placed and surprising
The action buttons row in HomeView (bottom-right) has: location pin · star · notifications · chatbot. The star opens an "Rate this App" dialog mid-session. Users tap it expecting a favorites or bookmark action (star = save in most apps). This causes confusion. The rating dialog also posts to /core/ratings/ which is a dead endpoint (table is redundant). Fix: Move app rating to Settings. Replace the star button with a "Favorites" bookmark icon that goes to /favorites.
⚠ Notification badge shown on both "Feedback" nav item and "Notifications" nav item simultaneously
In App.vue, unreadCount is fetched from /api/notifications/unread-count/ and displayed as a badge on both the /feedback nav item AND the separate /notifications nav link. Feedback and Notifications are different things — the badge should only appear on Notifications. Fix: Apply the badge only to the Notifications link. Use a separate newFeedback count (from /api/feedback/?unreviewed=true) if a Feedback badge is needed.
⚠ MapView and NavigateView have separate pan/zoom implementations — different feel
Despite both using useMapPanZoom, MapView.vue overrides onTouchStart and onTouchMove locally (the bug from Panel 1). NavigateView also overrides them. This means the pinch-to-zoom feel may be inconsistent between the two map screens. Fix: Use the composable exclusively — remove all local overrides.
✓ Slide-up bottom sheet for marker detail — correct mobile pattern
MapView uses a <transition name="slide-up"> bottom sheet for the selected marker detail. This is the right UX pattern for mobile — matches Google Maps behavior. The handle bar, close button, and action buttons are all in good positions.
✓ Dark mode — properly token-based via CSS custom properties
The design system uses --color-bg, --color-text-primary etc. throughout. Dark mode toggle works via themeStore and body class switching. No hardcoded hex colors in CSS (except a few legacy spots in HomeView inline styles).
✓ Offline banner — correct placement and messaging
"You are offline — map and navigation still work using saved data" is an excellent offline message. Specific, reassuring, not alarming.
💡 Missing: Map has no "You are here" or current building context indicator
When a user opens the map, there is no indication of where they currently are on campus. The "Set Location" dialog exists but is buried (manual text entry). On a small campus, a simple "I am at: [Main Gate / MST Building / etc.]" selector chip at the top of the map would massively improve the Navigate flow — the "From" field in NavigateView could auto-fill from this.
💡 Missing: Turn-by-turn directions should highlight the current step, not list all at once
NavigateView shows all steps in a scrollable list. For a campus walk, showing only the active step (Step 2 of 4) with a "Next" button matches how people actually navigate. The current list is fine for preview but the active navigation mode should be step-by-step.
💡 Missing: No loading skeletons — map markers and dropdowns just "pop in"
When data loads, markers and dropdown items appear instantly without transition. On a slow connection the screen looks empty for 1–2 seconds then fills suddenly. Adding skeleton placeholders (grey shimmer boxes where markers will appear) would make the app feel faster and more polished.
