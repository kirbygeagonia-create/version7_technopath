🔒 Security Issues
CRITICAL
JWT tokens stored in localStorage — XSS vulnerable
frontend/src/services/api.js · stores/authStore.js
▸
Access tokens and refresh tokens are stored in localStorage, which is readable by any JavaScript running on the page. A single XSS injection could steal admin tokens and give an attacker full admin access.
Fix: Store tokens in HttpOnly cookies set by the Django backend, not in localStorage. Then Django handles CSRF protection automatically.
CRITICAL
No rate limiting on login endpoint — brute-force possible beyond lockout
backend_django/technopath/settings.py · apps/users/views.py
▸
Django REST Framework has no DEFAULT_THROTTLE_CLASSES configured anywhere. While there is a 5-attempt lockout per account, an attacker can spray credentials across many accounts simultaneously with no server-side rate limit.
Fix: Add AnonRateThrottle and UserRateThrottle in DRF settings. Also add Django's django-axes for IP-based lockout.
CRITICAL
Default insecure SECRET_KEY in settings.py — committed to repo
backend_django/technopath/settings.py line 5
▸
The fallback default value is 'django-insecure-technopath-seait-dev-key-change-in-production'. If deployed without a proper .env file, this known key is used in production — compromising all HMAC signatures including JWT tokens.
Fix: Remove the fallback entirely: SECRET_KEY = config('SECRET_KEY') with no default. This forces a crash if the env var is missing, preventing silent misconfiguration.
HIGH
Chatbot Flask app has duplicate OpenAI client initialization
chatbot_flask/app.py lines 7–27
▸
The Flask chatbot creates an OpenAI client via the new SDK on line 14 (client = OpenAI(...)), then immediately creates a second one using the legacy openai.api_key pattern on lines 21–26. The first client object is used in the actual chat function. The second block is dead code and suggests copy-paste confusion.
Fix: Remove the second legacy block entirely. Keep only the client = OpenAI(api_key=os.getenv('OPENAI_API_KEY')) pattern and guard with if not key: raise RuntimeError().
HIGH
No chatbot rate limiting — OpenAI API cost exposure
chatbot_flask/app.py
▸
The Flask chatbot's /chat endpoint has no rate limiting. Any user can spam the endpoint, triggering unlimited OpenAI API calls and running up costs with no safeguard.
Fix: Add flask-limiter with per-IP limits (e.g. 20 requests/minute). Also add message length validation before calling the OpenAI API.
⚙ Backend Issues
CRITICAL
AuditLog filters applied after QuerySet slice — silently broken
backend_django/apps/users/views.py · AuditLogView.get()
▸
The queryset is sliced to 300 rows before filters are applied: 
qs = AdminAuditLog.objects.select_related('admin').order_by('-created_at')[:300]
if request.query_params.get('entity_type'):
    qs = qs.filter(entity_type=...)
In Django, calling .filter() on a sliced QuerySet raises a TypeError at runtime. The admin audit log filter UI will crash for any filter query.
Fix: Apply all filters before slicing. 
qs = AdminAuditLog.objects.select_related('admin').order_by('-created_at')
if ...: qs = qs.filter(...)
qs = qs[:300]
CRITICAL
CanManageRoom.has_object_permission() references wrong attribute
backend_django/apps/users/permissions.py line 63
▸
The permission check does: 
return obj.facility.department.code == request.user.department
But Facility.department is a ForeignKey to core.Department. The Department model has a code field. This would work if the FK is resolved — but request.user.department stores the plain string code (e.g. 'college_ict') while Department.code is something like 'ICT'. These won't match, meaning non-super-admin users can never pass the object-level permission check.
Fix: Align the comparison. Either compare obj.facility.department.code to the correct format, or use the department FK id directly. Also add a null guard for obj.facility.department.
HIGH
PublicDirectoryView calls non-existent method get_role_label()
backend_django/apps/users/views.py · PublicDirectoryView
▸
The view serializes user data with 'role': user.get_role_label(), but AdminUser has no such method. The correct method is get_role_display() (Django's auto-generated choices display method).
Fix: Replace user.get_role_label() with user.get_role_display(). This endpoint will raise a 500 AttributeError every time it is called.
HIGH
django-filters not in requirements.txt — will crash on import
backend_django/requirements.txt · apps/core/views.py
▸
apps/core/views.py imports from django_filters.rest_framework import DjangoFilterBackend and is listed in INSTALLED_APPS as django_filters. However django-filter is absent from requirements.txt. A fresh deployment will fail to start with an ImportError.
Fix: Add django-filter>=23.0 to requirements.txt.
HIGH
Navigation delete uses hard-delete via DestroyAPIView instead of soft-delete
backend_django/apps/navigation/views.py
▸
NavigationNodeDetailView and NavigationEdgeDetailView extend RetrieveUpdateDestroyAPIView with no custom perform_destroy(). This means DELETE on a navigation node permanently removes it from the database, even though the model has an is_deleted field intended for soft delete. In contrast, Rooms correctly overrides perform_destroy to set is_deleted = True.
Fix: Add perform_destroy(self, instance): instance.is_deleted = True; instance.save() to both navigation detail views.
MEDIUM
Announcement.publish() called twice when SuperAdmin directly publishes
backend_django/apps/announcements/views.py · AnnouncementCreateView.post()
▸
When publishes_direct=True, the view sets status='published' before a.save(), then immediately calls a.publish() which sets the same fields again and creates a Notification. This is harmless but creates an unnecessary extra UPDATE SQL and is confusing — the model already has a clean publish() method meant for this.
Fix: For the direct-publish path, do not pre-set status before save. Let a.publish() handle it exclusively.
MEDIUM
SQLite in production — not suitable for concurrent use
backend_django/technopath/settings.py
▸
The database is configured as SQLite with no override path for production. SQLite cannot handle concurrent writes and will lock under any real multi-user load. It is fine for development but must not be deployed to production.
Fix: Add a separate production settings file that uses PostgreSQL. Use dj-database-url to read the DATABASE_URL environment variable.
🖥 Frontend Issues
CRITICAL
MapView.vue uses undefined variables panStart and showToast
frontend/src/views/MapView.vue · onTouchStart(), addFavorite()
▸
onTouchStart sets panStart.value = ... and reads panStart.value in onTouchMove, but panStart is never declared with ref() in the script. Similarly, addFavorite() calls showToast(...) which is never imported or defined. Both are runtime errors — touch panning will crash on mobile and favoriting will throw a ReferenceError.
Fix: Add const panStart = ref({ x: 0, y: 0 }), and import/define a showToast function. The toast.js service already exists in /services/toast.js.
HIGH
NavigateView route overlay uses fixed SVG viewBox — misaligned on real data
frontend/src/views/NavigateView.vue · route SVG overlay
▸
The route polyline SVG has a hardcoded viewBox="0 0 800 600" with preserveAspectRatio="none", and path coordinates are computed as node.x * 800, node.y * 600. But the actual SEAIT_Map.svg has its own natural dimensions. If the map renders at any other aspect ratio, all route points will be displaced from the buildings they reference.
Fix: The route SVG overlay should use the same viewBox as the map image dimensions, or use percentage-based CSS positioning for each waypoint node instead of a fixed SVG.
HIGH
NavigateView locationDropdown shows all facilities+rooms unsorted, no search
frontend/src/views/NavigateView.vue · loadData()
▸
Location selection uses a plain <select> dropdown loaded with the full flat list of all facilities and all rooms merged together. With enough data this becomes very hard to use on mobile. There is also no type-ahead or search.
Recommendation: Replace both selects with a searchable autocomplete input — the existing SearchBar.vue component should be reused here with type-ahead filtering.
HIGH
Mock hardcoded room data leaks into production builds
frontend/src/views/MapView.vue · useMockRooms()
▸
useMockRooms() contains hardcoded room data for MST, JST, and RST buildings and is called unconditionally whenever the API returns 0 rooms — regardless of whether it is a dev or production build. This means if a real building is created in the database with 0 rooms, the app shows stale hardcoded mock data to users in production.
Fix: Wrap the mock fallback in if (import.meta.env.DEV). In production, show an empty state message instead of mock data.
MEDIUM
Dijkstra distance calculation uses raw database units, not real meters
frontend/src/services/pathfinder.js · generateSteps()
▸
The step distance is computed as Math.sqrt(dx*dx + dy*dy) where dx = (curr.x - prev.x) * 200. This 200 scale factor is arbitrary and has no relationship to actual campus meters. The displayed distance (e.g. "~47m") can be meaninglessly inaccurate.
Fix: The NavigationEdge.distance field in the database is where accurate real-world distances should be stored (in meters). The frontend should use result.distance from Dijkstra directly, which sums edge weights.
MEDIUM
Router splash guard fires on every first visit — not just first install
frontend/src/router/index.js
▸
The splash screen guard checks sessionStorage.getItem('tp_splash_shown'). Since session storage is cleared whenever the browser tab is closed, the splash plays every single new tab/session, not just on first app install. This may frustrate returning users.
Fix: Use localStorage with a versioned key (e.g. tp_splash_v1) so the splash only plays once per device, not every session.
LOW
No error boundary — unhandled promise rejections silently fail
frontend/src/main.js · general
▸
Vue's global error handler (app.config.errorHandler) is not configured. Unhandled errors in async lifecycle hooks or watchers will silently swallow in production, making debugging very difficult.
Fix: Add app.config.errorHandler = (err, instance, info) => { console.error(err); showToast('An error occurred', 'error') } in main.js.
🗃 Database / Schema Issues
HIGH
Feedback and core Rating models are duplicates — schema split
apps/feedback/models.py · apps/core/models.py
▸
There are two separate models for user feedback: apps.feedback.Feedback (table feedback) and apps.core.Rating (table ratings). Both have rating, comment, category, facility, and room fields. Additionally there is a FeedbackFlag in core that references Rating instead of Feedback, making the flag system only work on one of the two tables.
Fix: Consolidate into a single Feedback model and migrate all data. Remove the duplicate Rating model from core.
MEDIUM
NavigationNode has no unique constraint on map_svg_id — pathfinder can't reliably look up nodes
backend_django/apps/navigation/models.py
▸
map_svg_id is used by the pathfinder to find nodes by SVG element name, but there is no unique=True constraint. Duplicate SVG IDs can be created silently, and the pathfinder's .find() will return the first match, potentially routing to the wrong node.
Fix: Add unique=True to map_svg_id (with blank=True, null=True to allow nulls for waypoints that have no SVG element) and add a database migration.
MEDIUM
AppUsage and UsageAnalytics models are never populated — dead schema
backend_django/apps/core/models.py
▸
The database has app_usage, usage_analytics, device_preferences, connectivity_log, and search_history tables defined in core models but no corresponding API views, serializers, or frontend code that writes to them. These are schema stubs that add migration complexity with no current value.
Recommendation: Either implement the analytics pipeline or remove these models until needed. Unused schema creates confusion during code reviews.
LOW
NotificationReadStatus only works for admin users — public users' read state lost
backend_django/apps/notifications/models.py
▸
NotificationReadStatus has a FK to AdminUser. The comment says "Guest users track read status locally in IndexedDB." This means if a user clears their browser data, all read/unread notification state is permanently lost. A notification badge count may be permanently wrong.
Recommendation: Use a device-fingerprint or anonymous session ID to persist read state more reliably, or accept this as a known limitation and document it.
✓ What Works Well
Dijkstra implementationBinary heap priority queue is correct and efficient. Normalization helpers handle field name mismatches gracefully.
Role-based permissionsThe multi-role system (super_admin, dean, program_head, basic_ed_head) is well-thought-out with clear permission delegation and object-level checks.
Offline-first architectureIndexedDB caching with stale-check logic and network-first/cache-fallback pattern is the correct approach for a campus app.
Announcement workflowThe approval pipeline (pending → published/rejected → archived) with role-scoped publishing rules is well-designed and properly modeled.
JWT refresh with queueThe token refresh logic in api.js correctly queues concurrent requests while refreshing, preventing multiple simultaneous refresh calls.
Soft delete everywhereConsistent use of is_deleted flags across facilities, rooms, and navigation data preserves history. (Except navigation views — see above.)
Audit loggingThe write_audit() helper and AdminAuditLog model provide good traceability for all admin actions with IP capture.
PWA configurationVite PWA plugin is properly configured with service worker, manifest, and NetworkFirst caching strategy for the API.
