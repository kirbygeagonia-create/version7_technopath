🔄 Sync Engine — services/sync.js
CRITICAL
sync.js writes to db.notification_types — table was deleted in db.js v5
frontend/src/services/sync.js lines 87, 115–116 · src/services/db.js v5
▸
The db.js schema comment explicitly lists notification_types as a table that was removed: "Removed: notification_types, notification_preferences, users...". Yet syncAllData() still fetches /core/notification-types/, includes db.notification_types in the transaction, and calls db.notification_types.clear() and db.notification_types.bulkPut(). 
await db.transaction('rw',
  ...
  db.notification_types,  // ← this table no longer exists
  ...
Dexie will throw a TypeError: Cannot read properties of undefined when the sync runs, causing the entire sync to fail — no data at all gets cached for offline use.
Fix: Remove the notificationTypesRes fetch, the db.notification_types transaction entry, and the two write lines from syncAllData().
CRITICAL
syncAllData() is all-or-nothing — one failing API endpoint breaks all offline caching
frontend/src/services/sync.js · Promise.all()
▸
All 11 API calls are wrapped in a single Promise.all(). If any one endpoint returns an error (e.g. the /core/app-config/ endpoint is slow or unavailable), the entire Promise.all rejects, the catch block fires, the IndexedDB transaction never runs, and the app has zero cached data for offline use. A single flaky endpoint starves the entire offline cache.
Fix: Fetch each resource individually with its own try/catch, or use Promise.allSettled() and only write the successfully-fetched items. This way a single broken endpoint doesn't block the others.
HIGH
Dashboard calls /core/audit-log/ — URL does not exist; correct URL is /users/audit-log/
frontend/src/components/admin/AdminDashboard.vue line 297
▸
AdminDashboard fetches recent activity from api.get('/core/audit-log/?limit=5'). But the audit log is registered under the users app: path('api/users/', include('apps.users.urls')) → path('audit-log/', AuditLogView), meaning the correct URL is /api/users/audit-log/. The /api/core/ prefix has a different AdminAuditLogListView (a duplicate) under path('admin-audit-logs/', ...). The dashboard is calling neither of these correctly — it will get a 404 every load.
Fix: Change the dashboard fetch to api.get('/users/audit-log/'), or consolidate the two audit log endpoints into one.
MEDIUM
Polling replaces all notifications every 30 seconds — read state survives but badge count races
frontend/src/services/sync.js · startPolling()
▸
Every 30 seconds the poller clears and replaces the entire notifications table. It preserves read state via a read-state map, but any notification that arrives between the clear() and bulkPut() in a concurrent read will temporarily disappear. More importantly, if the user is actively viewing the notifications page during a poll cycle, the UI may flicker or lose its selected item.
Recommendation: Use an upsert/merge strategy instead of clear+replace. Add only new notifications (by id) and update existing ones only if changed.
🖥 Admin UI — Field & API Mismatches
CRITICAL
AdminNavGraph creates edges using node_id string — backend expects integer primary key
frontend/src/components/admin/AdminNavGraph.vue line 529 · createEdge()
▸
When the admin connects two nodes in the canvas, the connect-mode click handler calls: 
createEdge(connectStartNode.value.node_id, clickedNode.node_id)
Where node_id is the local string identifier like "ROOM_101". But createEdge() posts to /navigation/edges/ with: 
{ from_node: fromNodeId, to_node: toNodeId, ... }
Django's NavigationEdge model has from_node and to_node as integer ForeignKey fields. Sending a string like "ROOM_101" will cause a Django ValidationError. Every edge created via the canvas UI will fail silently (the error is only console.error'd).
Fix: Pass connectStartNode.value.id and clickedNode.id (the database integer PK) to createEdge(), not node_id.
HIGH
AdminFacilities filters and stats by facility.type — field does not exist in backend model
frontend/src/components/admin/AdminFacilities.vue · Facility model
▸
The admin facilities component filters by f.type (academic, admin, facility, library, canteen) and renders a type badge with facility.type. But the Facility model has no type field at all — it has code, department, description, and total_floors, but nothing called type. 
facilities.filter(f => f.type === 'academic').length  // → always 0
All three stat counters (Academic, Administrative, Facilities) will always show 0. The type filter dropdown will never match anything. Type badges will be blank.
Fix: Either add a facility_type field to the Facility model with the appropriate choices, or remove the type-based filtering from the admin UI and replace with department-based grouping.
HIGH
AdminNavGraph canvas renders node.node_id label — backend field is map_svg_id not node_id
frontend/src/components/admin/AdminNavGraph.vue lines 98, 415
▸
The canvas draws labels from node.node_id and the table shows node.node_id. The NavigationNode model has no node_id field — the corresponding field is map_svg_id. The nodeForm also has a node_id input that maps to this phantom field. Everything displayed as the node identifier in the graph editor will be undefined.
Fix: Rename all references from node_id to map_svg_id throughout AdminNavGraph.vue. Also update the nodeForm field name and the canvas label rendering.
HIGH
AdminNavGraph edge lookup uses node.node_id but edges store integer IDs — graph draws no connections
frontend/src/components/admin/AdminNavGraph.vue lines 360–361
▸
When drawing edges on the canvas: 
const fromNode = nodes.value.find(n => n.node_id === edge.from)
const toNode   = nodes.value.find(n => n.node_id === edge.to)
But edges from the API have from_node_id and to_node_id (integer FKs). So edge.from and edge.to are undefined. Both find() calls return undefined, and no lines are drawn between nodes. The graph canvas will show nodes floating in space with no visible connections at all.
Fix: Change the lookup to use the correct serializer field names: nodes.value.find(n => n.id === edge.from_node_id) and n.id === edge.to_node_id.
MEDIUM
AdminNavGraph saveNode sends full nodeForm including phantom node_id field to backend
frontend/src/components/admin/AdminNavGraph.vue · saveNode()
▸
saveNode() calls api.put('/navigation/nodes/...', normalizedNode) where normalizedNode is a spread of the full nodeForm object which includes node_id, type, and building fields. The backend serializer accepts only name, node_type, map_svg_id, x, y, floor, facility, room. The extra fields are ignored by DRF, but the correct fields are missing — specifically node_type (sent as type) so node type is never saved.
Fix: Map form fields to backend field names before POST/PUT. The form's type → node_type, node_id → map_svg_id.
🔒 Privacy & Access Control
CRITICAL
FeedbackListView is publicly readable — anyone can read all feedback including non-anonymous submissions
backend_django/apps/feedback/views.py · FeedbackListView
▸
FeedbackListView uses permission_classes = [IsAuthenticatedOrReadOnly], meaning any unauthenticated user can do a GET to /api/feedback/ and read all feedback records. The serializer uses fields = '__all__' so every field including is_anonymous, comment, and location is exposed. A student who submits feedback believing it is anonymous can have it read by anyone with API access.
Fix: Change permission to IsAuthenticated (admin-only read), and create a separate write-only public endpoint for submitting feedback. For admin reads, filter based on role: super admin sees all, dean sees only department-scoped feedback.
HIGH
FeedbackDetailView allows unauthenticated PATCH/DELETE on any feedback record
backend_django/apps/feedback/views.py · FeedbackDetailView
▸
FeedbackDetailView also uses IsAuthenticatedOrReadOnly. The IsAuthenticatedOrReadOnly permission class allows unsafe methods (PUT, PATCH, DELETE) only when authenticated. However since any admin token grants full access, there is no object-level check — a dean can delete another department's feedback, or a program head can modify feedback not related to their area.
Fix: Use IsSuperAdmin for delete, and for the admin role who submitted the feedback for read/edit access. Add object-level permissions.
HIGH
HomeView implements its own pan/zoom instead of using useMapPanZoom composable
frontend/src/views/HomeView.vue
▸
MapView.vue and NavigateView.vue correctly use the useMapPanZoom composable. But HomeView.vue has its own duplicate pan/zoom implementation with startPan, handlePan, startTouchPan, handleTouchPan, and handleZoom — all re-implemented from scratch (~60 lines of duplicated logic). Any bug fix or enhancement to pan/zoom must be done in two places.
Fix: Replace HomeView's inline pan/zoom implementation with useMapPanZoom(). This is a pure refactor — zero behaviour change, but eliminates the duplication.
⚙ Backend — Additional Issues
HIGH
Two separate AuditLog endpoints exist — /users/audit-log/ and /core/admin-audit-logs/ — returning different data
backend_django/apps/users/urls.py · apps/core/urls.py
▸
The users app has AuditLogView at /api/users/audit-log/ (used by AdminAuditLog view in the frontend). Core app has a separate AdminAuditLogListView at /api/core/admin-audit-logs/ using a different serializer. They read from the same table but different views, filters, and permission logic. This creates confusion about which endpoint is authoritative and may return inconsistent results depending on which one is called.
Fix: Delete one of the two endpoints. Keep /api/users/audit-log/ as it has the role-permission check, and remove the core duplicate.
MEDIUM
ChatbotView hardcodes Flask URL to localhost — fails in any deployed environment
frontend/src/services/aiChatbot.js line 1 · VITE_FLASK_CHATBOT_URL
▸
The Flask chatbot URL defaults to http://localhost:5000 when VITE_FLASK_CHATBOT_URL is not set. There is no Vite proxy configured for the Flask endpoint (only /api is proxied to Django). In any deployed or staging environment without the env variable, the chatbot will make cross-origin requests to localhost:5000 which will fail. The Flask app has no ALLOWED_HOSTS-equivalent for production domains.
Fix: Add the Flask chatbot to the Vite proxy config under a path like /chatbot-api pointing to the Flask server. Add the production domain to Flask CORS origins.
MEDIUM
Chatbot has no message length cap — very long inputs sent directly to OpenAI
chatbot_flask/app.py · /chat endpoint
▸
After checking for an empty message, there is no maximum length validation. A user can send a 50,000-character message that gets passed directly to OpenAI, consuming a large number of tokens and potentially triggering a context-length error from the model.
Fix: Add if len(message) > 1000: return jsonify({"error": "Message too long"}), 400 after the empty check. Also truncate conversation history tokens before sending.
LOW
NavigationNodeSerializer exposes both x/y and x_position/y_position — redundant fields
backend_django/apps/navigation/serializers.py
▸
The serializer adds x_position and y_position as read-only aliases for x and y. This is a forward-compat shim, but it means the API response contains both x: 0.3 and x_position: 0.3. The pathfinder's normalizeNode() uses ?? chains to handle both. This dual-field approach adds confusion — pick one and stick to it.
Recommendation: Standardize on x and y in the model and API. Update the pathfinder to only read x and y. Remove the aliases once the frontend is updated.
